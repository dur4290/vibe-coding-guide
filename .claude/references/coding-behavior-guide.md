# 코딩 작업 행동 기준

코딩 작업에서 흔한 실수를 줄이기 위한 기준입니다.
프로젝트별 지시와 함께 사용하되, 학생용 워크스페이스에서는 속도보다 안정성과 이해를 우선합니다.

## 1. 만들기 전에 생각하기

- 확실하지 않은 내용을 추측하지 않습니다.
- 애매한 점이 있으면 먼저 무엇이 헷갈리는지 말하고 질문합니다.
- 해석이 여러 개면 가능한 선택지를 짧게 보여줍니다.
- 더 단순한 방법이 있으면 먼저 제안합니다.
- 사용자가 요청한 방향이 위험하거나 너무 복잡하면 이유를 설명하고 더 작은 방법을 제안합니다.

## 2. 단순하게 만들기

- 요청받은 문제를 해결하는 최소 코드부터 만듭니다.
- 요청받지 않은 기능을 미리 추가하지 않습니다.
- 한 번만 쓰는 코드를 불필요하게 함수나 클래스로 나누지 않습니다.
- 실제로 일어날 가능성이 낮은 상황까지 과하게 처리하지 않습니다.
- 코드가 길어졌다면 더 짧고 쉬운 방법이 있는지 다시 봅니다.

## 3. 필요한 부분만 고치기

- 기존 파일을 수정할 때는 요청과 직접 관련된 줄만 바꿉니다.
- 주변 코드, 주석, 포맷을 이유 없이 정리하지 않습니다.
- 기존 스타일을 따릅니다.
- 관련 없는 죽은 코드나 이상한 부분을 발견하면 삭제하지 말고 알려줍니다.
- 내가 만든 변경 때문에 불필요해진 import, 변수, 함수는 정리합니다.

## 4. 목표와 확인 방법을 정하기

작업을 시작하기 전에 성공 기준을 짧게 잡습니다.

예시:

- "버그 수정" → "오류가 재현되는 입력을 확인하고, 수정 후 같은 입력에서 정상 실행되는지 확인"
- "기능 추가" → "입력 예시를 만들고, 결과 파일이 생성되는지 확인"
- "코드 정리" → "정리 전후 실행 결과가 같은지 확인"

여러 단계 작업이면 아래처럼 진행합니다.

```text
1. 입력/출력 확인 → 확인: 예시 입력이 준비됨
2. 최소 버전 작성 → 확인: 한 번 실행됨
3. 오류 처리 추가 → 확인: 잘못된 입력에서도 안내 메시지가 나옴
```

## 5. 학생에게 설명하기

- 코드만 던지지 말고 무엇이 만들어졌는지 설명합니다.
- 실행 방법을 알려줍니다.
- 성공하면 어떤 결과가 보여야 하는지 알려줍니다.
- 다음에 할 일을 1개만 명확하게 제안합니다.

## 좋은 작업의 기준

- 변경 diff가 작고 이유가 분명합니다.
- 학생이 파일을 열었을 때 다음 행동을 알 수 있습니다.
- 코드는 작게 시작하고, 확인하면서 자랍니다.
- 애매한 요청은 구현 전에 질문합니다.
